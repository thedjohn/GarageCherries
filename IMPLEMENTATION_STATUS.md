Root cause confirmed via a temporary production debug log (added, checked, then reverted same session): both routes fetched raw `listing_views` rows for a batch of listings (`.select('listing_id').in('listing_id', carIds)`) and counted them in JS, with no `.limit()`/pagination — but Supabase/PostgREST caps any unbounded `.select()` at 1000 rows by default. Verified directly: the exact same query for 13 real listings returned exactly 1000 rows, while the true total (`count: 'exact'`) was 1293 — 293 real view records were being silently dropped, undercounting whichever listings' rows happened to fall outside the first 1000 returned (no `ORDER BY`, so arbitrary). Fixed by moving the counting into the database via a new `count_listing_views(p_listing_ids)` Postgres function (`GROUP BY listing_id, COUNT(*)`, migration `supabase/migrations/20260719_count_listing_views_rpc.sql`, run manually per project convention) and calling it via `.rpc()` instead of fetching raw rows — Postgres aggregates correctly regardless of row volume. Verified end-to-end against live production data: the RPC's per-listing sum now exactly matches the true total (1293), and the specific listing that was undercounted (8→9) now returns the correct count. `totalWatchers` (watchlist counts) left unchanged in both routes — same unbounded-fetch pattern, but not confirmed broken since watchlist batches are far less likely to cross 1000 rows in practice. The same latent pattern also exists in the dealer digest email and today's `/reports` fix, scoped differently (by dealer or by date) — not confirmed broken, treated as a separate future item, not bundled into this fix (fixed 2026-07-19)
- [x] **Advertiser public profile pages were 404ing for every real visitor — fixed via an RLS policy, no code change needed** — this is the new `GET /api/cron/sitemap-health` job's very first real catch, less than a day after shipping: it emailed an alert for `https://www.garagecherries.com/advertisers/mobile-auto-pros` returning 404. Traced to a missing/broken Row Level Security policy on the `advertisers` table — confirmed directly that querying with the anon key (what real visitors and Googlebot actually use, as opposed to the service-role key used everywhere in this codebase for diagnostics) returned zero rows for a genuinely active, unexpired advertiser, while `app/advertisers/[slug]/page.tsx` correctly falls through to `notFound()` whenever its query comes back empty. Since this is currently the *only* active advertiser in the database, this likely means the entire advertiser public-profile feature has never actually rendered for a real anonymous visitor — internal testing presumably happened while authenticated, which may hit different RLS rules and would fully mask an anon-only block. Fixed by adding `create policy "Public can view active advertisers" on advertisers for select to anon, authenticated using (active = true)` (run manually in Supabase's SQL Editor). Verified after the policy was applied: the anon key now reads the row correctly, and `https://www.garagecherries.com/advertisers/mobile-auto-pros` returns a real 200 with the business name rendering. Pure database fix — no application code changed (fixed 2026-07-19)
